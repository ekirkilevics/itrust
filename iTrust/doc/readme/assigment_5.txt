	Update Feature Design
		We chose to update UC22. Our updated use case can be found in our requirements document (See link above). 
		UC22 is the scheduling an appointment use case. We felt that it was stupid to be able 
			to create an appointment but not edit or delete said appointment. If the HCP accidentally 
			entered the wrong date or time, there needed to be a way to change it. If a patient 
			had an emergency come up and had to cancel or move an appointment, there was no way 
			to do that without manually editing the database.Now, a patient can contact their HCP
			and the HCP can edit the date of the appointment. With our ability to edit and delete
			future appointments, HCP's can know that their calendars won't have extraneous appointments
			or appointments at the wrong time. 
		
Design Decisions:
	We decided to strip out most of the coding for iTrust and implement a color scheme based on user preferences.
		To do this, we gave the Admin the ability to set a global/default color scheme. We also gave users 
		the ability to choose their own color, which is saved from session to session.
	We decided to refactor the logging. We entirely stripped out the previous logging implementation
		and moved the logging into the jsp layer instead. We chose to do this because with the logging in the
		action layer, when a page was pulled up sometimes it would call the action multiple times for one view
		creating extra logging. Moving the logging to the jsp layer eliminates that redundancy. We also refactored
		the transaction codes. More information on those decisions can be found at:
		http://ada.csc.ncsu.edu/wiki/doku.php?id=team6:log_codes#section199. We also discovered we could not log
		when users get logged out due to inactivity. 
	To implement recaptcha, we had the login page process some logic before sending it off to jsecuritycheck. 
		This means that if the user gets the captcha wrong it won't bother processing the login information. 
	We added two tables to implement the override reason codes when an HCP is prescribing something that will
		cause a reaction. These tables are OVReactionOverride and DrugReactionOverrideCodes. 
	We decided to remove the information item on the top of the page. It showed the user their own information,
		which seems rather silly since the patient should know who they are and likely gave that information 
		to the HCP. Also, the patient can view that under edit my demographics if they aren't sure.
